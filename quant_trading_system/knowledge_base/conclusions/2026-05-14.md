## Observations and reasoning

**The infrastructure side is fully back; the open anomaly is the only
remaining blocker.** Day 5 surfaced two problems on top of the resolved
4-day proxy outage: an unexplained portfolio and a SIP-subscription wall
on the bars feed. The feed issue was patched (post-run) yesterday by
threading an `ALPACA_DATA_FEED=iex` default through
`market_data_service.py`, and today's `regime`, `bars`, and `quote`
calls all confirm that patch holds — regime number is bit-identical to
yesterday's verification, and `quote SPY` even returned a two-sided
post-hours quote today versus the zero-ask anomaly that surfaced once
yesterday. Net: the only operational problem on day 6 is the same one
yesterday flagged in the loud `ATTENTION.md` — a real long book of 10
unattributed positions sitting in the paper account. Until the operator
either (a) flat-closes those positions or (b) tells the harness which
strategy id to tag them under, the journal-driven workflow the harness
is built around cannot start cleanly.

**Per yesterday's explicit recommendation, "do nothing" is again the
correct call.** Yesterday's handoff item 1 spells out exactly this
branch: *if the 10 positions are still there and there's no operator
note anywhere, treat them the same way I did today: don't close them,
don't tag them, document and stop. The risk profile of flattening
someone's intentional book is much worse than letting it sit one more
day.* That is precisely today's situation. The diff vs. yesterday is
essentially nil:

  - Same 10 positions, same avg-entry prices, same lots, slightly
    different marks.
  - Same `ATTENTION.md` (mtime unchanged from when I wrote it
    yesterday at 17:44).
  - Same library on disk (18 active, 0 archived).
  - Same empty journal.
  - Same `runs/` directory (only the day-1 auth artifact).
  - Same regime (bull, 0.76 conf), same `get-active` (empty).

The decision tree therefore lands in the same leaf: do not call
`set-active`, do not submit any orders, do not modify any strategy
file, do not touch the 10 positions. Yesterday-me anticipated this
exact case and wrote the instruction explicitly; following it is the
right call.

**Why I did *not* run the "real day-1 workflow" today even though the
regime is now classifiable.** Yesterday's handoff item 2 conditionally
permitted that path ("if regime is still bull/trending you can run the
*real* day-1 workflow"), but it's gated on the unexplained portfolio
being resolved (item 1). It isn't. Two specific reasons reinforce that
gating:

  1. **Buying power is only $15,948.** With ~$208k of equity already
     deployed against the manual book, any meaningfully-sized harness
     trade would either crowd the manual book or use a tiny slice of
     equity (sub-1% of portfolio value) that's not really worth the
     accounting overhead. The `MAX_POSITION_SIZE_PCT` cap is computed
     against equity ($112k), so a 10%-capped position is up to ~$11k
     — that's nearly 70% of available buying power and would burn
     margin headroom on a single trade.
  2. **Journal corruption risk.** If I call `set-active` today and then
     submit even one tagged order, tomorrow's `portfolio-health` will
     attribute that strategy's P&L while still being unable to
     account for the 10 manual positions sitting on the same broker
     account. The two P&L streams would mix in any account-level
     measurement (equity curve, drawdown, etc.), and the per-strategy
     metrics that *should* be isolated would only stay isolated until
     someone runs an account-level health calculation. The cleanest
     record is the one where the harness's first journal entry
     postdates a clean baseline — either the manual book closed, or
     the manual book formally assigned a strategy id.

**Why I did *not* edit `active_strategy.md` or any strategy file.** Same
reasoning as days 1–5: no journal history, no health snapshot to drive
a parameter choice, and the active-strategy decision is gated on the
unexplained portfolio. Setting an active strategy today would either
be a guess (no health data) or a post-hoc rationalisation of someone
else's book — both bad for the durable record.

**Why I did *not* refresh `ATTENTION.md`.** The file is still accurate.
